This lab is the first in the **Model Compression** series. The series follows a single roadmap: take any PyTorch LLM or VLM, compress it, and ship it — whether that means a server GPU or an edge device like a Qualcomm chip. The full path looks like this: **train or download** a baseline FP16 model → **quantize** weights (and optionally activations) to a lower-bit format → **save** the quantized checkpoint in a format the target runtime understands → **deploy** using hardware-accelerated kernels (CUDA for NVIDIA GPUs, QNN/SNPE for Qualcomm). Every step has pitfalls: a flawed checkpoint format breaks the load path, a naive dequant loop erases all latency gains, a missing calibration dataset inflates perplexity. The series walks you through each step from scratch so you understand exactly what production tools like AWQ, AutoGPTQ, and llm-compressor are doing under the hood — and why they make the design choices they do.


## Lab 1: W4A16 from Scratch


This lab builds a minimal but correct W4A16 quantization layer in pure PyTorch — no library magic, no fused kernels yet. The goal is to hit two failure modes deliberately: a load-path crash caused by a flawed constructor design, and a 16× latency regression caused by CPU dequantization in the forward pass. By the end you will have a working `WQLinear` module that saves and loads cleanly, understand exactly why `register_buffer` is the right primitive for quantized weights, and have a benchmark baseline (PPL, latency) to improve on in the next lab when we add fused kernels and calibration.

The complete code implementation can be found in the notebook: [w4a16_scratch.ipynb](https://github.com/hungmanh21/hungmanh21.github.io/tree/main/notebooks/w4a16_scratch.ipynb)

## 1. The Naive Implementation — and Its Design Flaw


The most naive approach you can imagine: load the model, loop over its layers, quantize each weight tensor in-place, and save.


```python
for name, module in model.named_modules():
    if isinstance(module, torch.nn.Linear):
        W = module.weight.data.float()
        w_q, scale = quantize_sym(W, group_size=128)   # returns int tensor + scale
        module.weight.data = w_q.to(torch.float16)     # assign quantized values back
model.save_pretrained("model_q")
```


This code runs without errors and produces a checkpoint. But it has several silent flaws:


1. **No memory saving — and you cannot fix it by assigning `uint8`.** `module.weight` is an `nn.Parameter`, and `nn.Parameter` enforces a floating-point dtype. Trying `module.weight.data = w_q.to(torch.uint8)` raises a `RuntimeError` at assignment time. Even if you got past that, `F.linear` would crash: matrix multiplication is not defined for integer dtypes in PyTorch. The only way to store weights in a non-float dtype is through `register_buffer`, which is not an `nn.Parameter` at all — it is a separate tracking mechanism that bypasses both constraints.
2. **Scale discarded.** `scale` is computed but never stored anywhere. There is no way to dequantize at inference time, so the output of every quantized layer is already corrupted.
3. **Load path identical to FP16.** The saved checkpoint has the same key names and dtypes as the original model. A vanilla `from_pretrained` loads it without complaint and runs "successfully" — while silently producing garbage logits, with no memory savings.


The fix is to actually change the storage format. Weights must be converted to `uint8` (packed nibbles) and stored as `register_buffer`, and scales must live alongside them as their own registered buffer. That requires a custom `nn.Module` subclass — not an in-place edit of `nn.Linear`.


The next step up is to write that subclass and swap every `nn.Linear` with it at quantization time:


```python
import torch
import torch.nn.functional as F
import copy


class W4A16Linear(torch.nn.Module):
    def __init__(self, linear: torch.nn.Linear, group_size: int = 128):
        super().__init__()
        W = linear.weight.data.float()
        self.group_size  = group_size
        self.out_shape   = W.shape


        w_q, scale = quantize_sym(W, group_size)


        self.register_buffer("w_int4", w_q)
        self.register_buffer("scale", scale)
        self.bias = linear.bias


    def forward(self, x):
        w = self.w_int4.float() * self.scale.float()
        w = w.reshape(self.out_shape)
        return F.linear(x, w.to(x.dtype), self.bias)




def replace_linear_layers(model: torch.nn.Module, **kwargs) -> torch.nn.Module:
    model = copy.deepcopy(model)
    for name, module in model.named_children():
        if isinstance(module, torch.nn.Linear):
            setattr(model, name, W4A16Linear(module, **kwargs))
        else:
            replace_linear_layers(module, **kwargs)
    return model
```


This works fine for one-shot quantization — quantize once, run inference. The problem surfaces at **save/load time**. During load, you build the model from config (skeleton only, no weights), then call `load_checkpoint_and_dispatch`. At that point there is no `nn.Linear` with a materialized weight tensor to pass into `W4A16Linear.__init__`. You either load the FP16 weights first (doubling peak VRAM: FP16 + INT4 co-exist in memory) or the constructor crashes.


> **Rule.** Quantized layer constructors must accept only *dimensions and metadata* — never a materialized weight tensor. Quantization logic belongs in a `from_linear` classmethod, not `__init__`.
{: .prompt-tip }


The fix is `WQLinear`: `__init__` takes only integers (dimensions) and allocates empty buffer shapes. A separate `from_linear` classmethod holds all the quantization math and can be bypassed (`init_only=True`) on the load path.


```python
import torch
import torch.nn as nn


class WQLinear(nn.Module):
    def __init__(self, w_bit, group_size, in_features, out_features, bias):
        super().__init__()
        self.w_bit = w_bit
        self.group_size = group_size
        self.in_features = in_features
        self.out_features = out_features


        # 2 int4 values packed into 1 uint8  →  out_features // 2 columns
        self.register_buffer("qweight", torch.zeros(in_features, out_features // 2, dtype=torch.uint8))
        self.register_buffer("scales",  torch.zeros(in_features // group_size, out_features, dtype=torch.float16))
        if bias:
            self.register_buffer("bias", torch.zeros(out_features, dtype=torch.float16))
        else:
            self.bias = None
```


<details>
<summary><strong>Why not just assign <code>self.weight = qweight</code>?</strong></summary>
<div style="padding:12px 4px 4px;font-size:14px;line-height:1.7;color:#1e293b;">


<p>It looks like it works — until it doesn't. Three things silently break:</p>


<p><strong>1. The tensor disappears on save.</strong> A plain attribute is not registered with PyTorch's module system. When you call <code>model.save_pretrained()</code> or <code>torch.save(model.state_dict())</code>, only <code>nn.Parameter</code>s and registered buffers are collected. <code>self.qweight</code> set as a plain attribute is invisible to <code>state_dict()</code> — you save an empty slot and load garbage.</p>


<p><strong>2. The tensor stays on CPU after <code>.to("cuda")</code>.</strong> <code>model.to("cuda")</code> only moves tensors that PyTorch knows about. A plain attribute is not tracked, so <code>qweight</code> stays on CPU while the rest of the model moves to GPU. The first <code>forward</code> call raises a device mismatch error — or silently computes on CPU if you accidentally keep everything there.</p>


<p><strong>3. <code>nn.Parameter</code> is not a fix.</strong> It solves the first two problems but adds a third: <code>nn.Parameter</code> requires a floating-point dtype and appears in <code>model.parameters()</code>, which means the optimizer tries to update it. <code>torch.uint8</code> is not a valid parameter dtype — the constructor throws. Even if you cast to float, the packed shape <code>(in_features, out_features // 2)</code> is wrong for a weight matrix, and gradient updates on packed nibbles are meaningless.</p>


<p><code>register_buffer</code> is the exact intersection: saved to <code>state_dict()</code>, moved by <code>.to()</code>, excluded from <code>parameters()</code>, accepts any dtype and any shape. It is the only correct choice for non-trainable integer tensors.</p>
</div>
</details>

`register_buffer` hits the exact intersection the table shows: saved to checkpoint, moves with the module, excluded from the optimizer. A plain attribute loses the first two guarantees — `qweight` would be silently dropped on `model.save_pretrained()`. An `nn.Parameter` adds gradient tracking and enforces float dtype, both of which break INT4: `torch.uint8` is not a valid `nn.Parameter` dtype, and the packed shape `(in_features, out_features // 2)` is incompatible with what the optimizer expects.


> **Note.** `register_buffer(..., persistent=False)` omits the tensor from `state_dict()` — useful for values you recompute at load time (e.g. running statistics in BatchNorm). INT4 weights must be `persistent=True` (the default).

## 3. Packing Two INT4 Values into One Byte


PyTorch has no native `int4` dtype — the smallest integer type it supports is `int8` (one byte). If you store each 4-bit weight in its own `int8` tensor, you waste the upper 4 bits of every byte and achieve zero memory reduction over a naive implementation. Packing two INT4 values into one `uint8` byte is what actually delivers the 2× storage saving: a weight matrix that would occupy 512 MB as `int8` shrinks to 256 MB packed. It also halves the amount of data the GPU has to load from VRAM during dequantization, which is why memory bandwidth — not compute — is the bottleneck on the load path.

The byte has 8 bit positions. The `& 0xF` mask (`0000 1111`) clamps any value to its low 4 bits before packing — without it, a value like 19 (`0001 0011`) would corrupt the high nibble. During unpack, `& 0xF` strips any residual bits above position 3 after the right shift.


The implementation also handles **sign extension** for symmetric INT4 (range $[-8, 7]$). After unpacking raw nibbles (0–15), values 8–15 are negative integers in disguise: nibble `1000` = 8 should be $-8$, `1111` = 15 should be $-1$. One line corrects this:


```python
out[out > 7] -= 16
```


Here is the full implementation:


```python
@staticmethod
def _pack(x_int):
    """(rows, cols) int32  →  (rows, cols//2) uint8  — 2 nibbles per byte"""
    return ((x_int[:, 0::2] & 0xF) | ((x_int[:, 1::2] & 0xF) << 4)).to(torch.uint8)


@staticmethod
def _unpack(packed):
    out = torch.zeros(packed.shape[0], packed.shape[1] * 2,
                      dtype=torch.int32, device=packed.device)
    out[:, 0::2] = packed.int() & 0xF
    out[:, 1::2] = (packed.int() >> 4) & 0xF
    # sign-extend: nibble values 8-15 mean -8 to -1
    out[out > 7] -= 16
    return out
```


`_pack` takes even columns (`0::2`) as the low nibble and odd columns (`1::2`) as the high nibble, ORs them together, and casts to `uint8`. The output is half the column count: `(rows, cols//2)`. `_unpack` reverses this exactly, then fixes the sign by subtracting 16 from any value above 7.

\text{pack}(lo, hi) = (lo \,\&\, \texttt{0xF}) \;\big|\; \bigl((hi \,\&\, \texttt{0xF}) \ll 4\bigr)
$$


$$
lo = \text{byte} \,\&\, \texttt{0xF}, \qquad hi = (\text{byte} \gg 4) \,\&\, \texttt{0xF}
$$

## 4. `forward()`: Dequantize on Demand


`forward` is the core of the module — this is what runs on every inference call. It undoes the packing from Section 3, scales each group back to float, and calls `F.linear`. The loop below runs on CPU, which is exactly why the benchmark in Section 8 shows a 16× latency regression.


```python
import torch.nn.functional as F


def forward(self, x):
    # Step 1 — unpack: (in, out//2) uint8 → (in, out) int32
    W_int = self._unpack(self.qweight)   # (in_features, out_features)


    # Step 2 — dequantize per group: W ≈ W_int * s
    W_fp = torch.empty(self.in_features, self.out_features,
                       dtype=torch.float32, device=self.qweight.device)
    for g in range(self.in_features // self.group_size):
        s  = self.scales[g].float()                                    # (out_features,)
        wg = W_int[g * self.group_size : (g + 1) * self.group_size].float()
        W_fp[g * self.group_size : (g + 1) * self.group_size] = wg * s


    # Step 3 — matmul: W_fp is (in, out), F.linear expects (out, in)
    return F.linear(x, W_fp.t().to(x.dtype), self.bias)
```


<div style="background:#f1f5f9;border:1px solid #d1d5db;border-left:3px solid #6366f1;border-radius:0 6px 6px 0;padding:14px 18px;margin:16px 0;font-family:system-ui,sans-serif;">
  <div style="font-size:10px;font-weight:700;text-transform:uppercase;letter-spacing:.07em;color:#6366f1;margin-bottom:8px;">Formula · Dequantization (Symmetric)</div>


$$
W \approx W_{\text{int}} \cdot s
$$


  <div style="font-size:12px;color:#64748b;margin-top:10px;line-height:1.7;">
    <strong>W<sub>int</sub></strong> — unpacked signed integer, range [−8, 7], from <code>_unpack(qweight)</code><br>
    <strong>s</strong> — per-group scale (float16), from <code>scales</code><br>
    Symmetric quantization has no zero point: the integer grid is centered at 0, so −8 maps to the most negative float and +7 to the most positive.
  </div>
</div>


> **Warning.** The dequant loop is a Python loop over CPU tensors. For a 4B model this runs across hundreds of groups per layer and dozens of layers — the GPU waits the entire time. This is the root cause of the 16× latency shown in Section 8. The fix is to fuse `_unpack` + dequant + matmul into a single CUDA kernel, covered in the next lab.
{: .prompt-warning }

## 5. `from_linear`: Quantizing Weights at Replacement Time


`from_linear` does two jobs: compute the quantization math, and build the replacement module. Keeping them together means the caller never touches raw weight tensors directly.


Before calling it, a calibration step computes per-group scales from the FP weight tensor. For each group of `group_size` rows along the input dimension:


```python
def compute_scales(W, group_size):
    """W : (in_features, out_features) float32
    Returns scales of shape (in_features // group_size, out_features) float16"""
    n_groups = W.shape[0] // group_size
    scales = torch.zeros(n_groups, W.shape[1])
    for g in range(n_groups):
        wg = W[g * group_size : (g + 1) * group_size]
        w_abs_max = wg.abs().max(dim=0).values   # (out_features,)
        scales[g] = w_abs_max / 7.0              # map [-w_max, w_max] onto [-7, 7]
    return scales.half()
```


<div style="background:#f1f5f9;border:1px solid #d1d5db;border-left:3px solid #6366f1;border-radius:0 6px 6px 0;padding:14px 18px;margin:16px 0;font-family:system-ui,sans-serif;">
  <div style="font-size:10px;font-weight:700;text-transform:uppercase;letter-spacing:.07em;color:#6366f1;margin-bottom:8px;">Formula · Symmetric Scale</div>


$$
s = \frac{\max|W|}{7}
$$


  <div style="font-size:12px;color:#64748b;margin-top:10px;line-height:1.7;">
    <strong>max|W|</strong> — largest absolute value in the group<br>
    <strong>7</strong> — the positive limit of signed INT4; using 7 (not 8) keeps the grid symmetric so 0.0 maps exactly to integer 0<br>
    No zero point is needed: the grid is centered at 0, which matches the roughly zero-centered distribution of LLM weight tensors.
  </div>
</div>


Then `from_linear` takes those precomputed scales, runs the forward quantization formula, and packs the result:


<div style="background:#f1f5f9;border:1px solid #d1d5db;border-left:3px solid #6366f1;border-radius:0 6px 6px 0;padding:14px 18px;margin:16px 0;font-family:system-ui,sans-serif;">
  <div style="font-size:10px;font-weight:700;text-transform:uppercase;letter-spacing:.07em;color:#6366f1;margin-bottom:8px;">Formula · Symmetric INT4 Quantization</div>


$$
W_{\text{int}} = \text{clamp}\!\left(\text{round}\!\left(\frac{W}{s}\right),\; -8,\; 7\right)
$$


</div>


```python
@classmethod
def from_linear(cls, linear, w_bit, group_size, scales, init_only=False):
    """
    linear    : nn.Linear being replaced
    scales    : (in_features // group_size, out_features)  float16  — from compute_scales
    init_only : True  → allocate empty buffers only (load path)
                False → run quantization math and fill buffers
    """
    q = cls(w_bit, group_size, linear.in_features, linear.out_features,
            bias=linear.bias is not None)


    if init_only:
        return q   # buffers will be filled from the checkpoint


    q.scales.copy_(scales.clone().half())


    W = linear.weight.data.float().t()  # (in_features, out_features)


    W_int = torch.zeros(linear.in_features, linear.out_features, dtype=torch.int8)


    for g in range(linear.in_features // group_size):
        s  = scales[g]                                          # (out_features,)
        wg = W[g * group_size : (g + 1) * group_size]          # (group_size, out_features)
        W_int[g * group_size : (g + 1) * group_size] = \
            torch.clamp(torch.round(wg / s), -8, 7).to(torch.int8)


    q.qweight.copy_(cls._pack(W_int))   # (in_features, out_features // 2) uint8


    if linear.bias is not None:
        q.bias.copy_(linear.bias.clone().half())


    return q
```


Each group of `group_size` rows shares one scale — **per-group quantization** — which is why `scales` has shape `(in_features // group_size, out_features)` rather than one scalar per layer. Grouping runs along the input dimension (rows) rather than the output dimension because `W` is transposed to `(in_features, out_features)` before quantization, so each row slice is a contiguous block in memory; fused dequant kernels then load one scale per group as they sweep down the input channels.


> **Tip.** Pass `init_only=True` on the load path. It skips all quantization math and returns a `WQLinear` with correctly-shaped zero-filled buffers, ready to receive weights from `load_checkpoint_and_dispatch`.
{: .prompt-tip }

## 6. Saving the Quantized Model


Once every `nn.Linear` is replaced by `WQLinear`, the model no longer holds any FP16 weight tensors — only `qweight` and `scales` buffers. `save_pretrained` calls `state_dict()` under the hood, and because those buffers are registered with `register_buffer`, they are all present and correctly named in the dict. Nothing extra is needed.


One important thing to understand: `save_pretrained` saves **only tensors** — it writes the `state_dict` to a `.safetensors` file and the architecture config to `config.json`. It does not save Python classes. The checkpoint has no record that the model uses `WQLinear` instead of `nn.Linear`. When you later call `from_pretrained`, Hugging Face reads `config.json` and reconstructs the model using whatever class the architecture maps to — which is vanilla `nn.Linear`. This is exactly why the load path requires two steps: you must replace the `nn.Linear` layers with `WQLinear` *before* filling them from the checkpoint, so the tensor key names (`qweight`, `qzeros`, `scales`) match what is in the file.


```python
# Replace all linear layers with quantized equivalents
model_q = replace_linear_with_wqlinear(model_fp, w_bit=4, group_size=128,
                                        scales_dict=scales_dict)


# Save — state_dict contains only qweight/scales, no FP16 weights
model_q.save_pretrained(save_dir)
tokenizer.save_pretrained(save_dir)


# Also write the quant config into config.json so the load path can reconstruct
# the WQLinear shapes without needing the original FP model
config = model_q.config
config.quantization_config = {"w_bit": 4, "group_size": 128}
config.save_pretrained(save_dir)
```

for name, module in list(model.named_modules()):
    if isinstance(module, nn.Linear):
        parent = get_parent(model, name)
        setattr(parent, name.split(".")[-1],
                WQLinear.from_linear(module,
                    w_bit=4, group_size=128,

                             device_map="auto")              # Step 3

### What is Perplexity?


**Perplexity (PPL)** measures how surprised a language model is by a held-out text. A perfect model that always predicts the next token with 100% confidence has PPL = 1. A model that is completely lost has PPL → ∞. Lower is better.


Formally, given a sequence of $N$ tokens $w_1, w_2, \ldots, w_N$:


$$
\text{PPL} = \exp\!\left(-\frac{1}{N} \sum_{i=1}^{N} \log P(w_i \mid w_1, \ldots, w_{i-1})\right)
$$


The inner sum is the total log-likelihood of the sequence under the model. Dividing by $N$ gives per-token average negative log-likelihood (NLL); exponentiating converts it back to a probability-scale number you can compare across sequence lengths.


**Concrete example.** Suppose the model processes 3 tokens and assigns these probabilities to the correct next token at each step:


| Step | Correct token | $P(\text{correct})$ | $\log P$ |
| ---- | ------------- | ------------------- | -------- |
| 1    | "the"         | 0.50                | −0.693   |
| 2    | "cat"         | 0.25                | −1.386   |
| 3    | "sat"         | 0.10                | −2.303   |


$$
\text{NLL} = -\frac{1}{3}(-0.693 - 1.386 - 2.303) = \frac{4.382}{3} = 1.461
$$


$$
\text{PPL} = e^{1.461} \approx 4.31
$$


Intuitively: the model is about as uncertain as if it were choosing uniformly among ~4 equally likely tokens at every step.


**Why Wikitext-2?** It is a standard, clean Wikipedia corpus with no data contamination concerns. Running PPL on Wikitext-2 takes a few minutes on GPU and gives a stable signal — a +1 PPL change is reproducible and meaningful. It is used on the *base* model (not instruct-tuned) because base models have not been RLHF-shifted away from the natural language distribution the benchmark assumes.


The PPL regression of +13.18% (8.935 → 10.113) is expected for naive uniform INT4 without calibration. Quantization rounds weights to the nearest grid point; without a calibration dataset to adjust scales toward the weight distribution, you lose more precision than necessary. AWQ and GPTQ both address this — covered in later labs.


The latency regression is not expected and is entirely a code artifact. The naive `forward` dequantizes INT4 → float32 on every call, which runs on CPU with no hardware acceleration. Every token generated pays a full dequantization pass across all weight tensors before the matmul even starts.


> **Warning.** Dequantizing inside `forward` moves the bottleneck from GPU matmul to CPU memory bandwidth. The INT4 model is *smaller* but *slower* because the savings in memory transfer are wiped out by the dequant overhead. The fix — fusing dequant + matmul into a single kernel — is covered in the next lab.
{: .prompt-warning }


The PPL and latency tell different stories: the quality cost (+13%) is a property of the quantization scheme and will improve with calibration (AWQ, GPTQ). The latency cost (16×) is a property of the implementation and goes to near-zero once the dequant is fused. Separating these two failure modes is the key takeaway from this lab.
