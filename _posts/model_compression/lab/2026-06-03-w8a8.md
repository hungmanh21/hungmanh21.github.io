W4A16 shrinks weights but leaves the matmul running in FP16. W8A8 quantizes **both** operands so the multiply itself runs on INT8 (or FP8) tensor cores — the path to actual throughput at large batch. This lab builds three W8A8 recipes from scratch on Qwen3-4B and watches one of them fail catastrophically before two others fix it. 

The complete code implementation can be found in the notebook: [w8a8.ipynb](https://github.com/hungmanh21/hungmanh21.github.io/tree/main/notebooks/w8a8.ipynb)

## 1. The W8A8 forward pass


*Why dequantize back to BF16 if the very next layer is just going to quantize again?*


<div style="background:#fffffc;border:1px solid #e2e8f0;border-radius:8px;padding:16px;margin:16px 0;font-family:system-ui,sans-serif;font-size:13px;">
  <div style="font-size:11px;color:#64748b;margin-bottom:10px;font-weight:600;">ONE W8A8 LINEAR — QUANTIZE → INT8 MATMUL → DEQUANT</div>
  <div style="display:flex;align-items:center;gap:6px;flex-wrap:wrap;">
    <div style="background:#a0c4ff;color:#1e3a8a;padding:8px 12px;border-radius:6px;border:1px solid #60a5fa;font-weight:600;">X &nbsp;BF16<div style="font-weight:400;font-size:10px;">[T × H]</div></div>
    <span style="color:#64748b;font-weight:700;">→</span>
    <div style="background:#fdffb6;color:#713f12;padding:8px 12px;border-radius:6px;border:1px solid #facc15;font-weight:600;">quantize<div style="font-weight:400;font-size:10px;">scale = max|X|/127</div></div>
    <span style="color:#64748b;font-weight:700;">→</span>
    <div style="background:#caffbf;color:#14532d;padding:8px 12px;border-radius:6px;border:1px solid #86efac;font-weight:600;">INT8 matmul<div style="font-weight:400;font-size:10px;">Xq · Wq</div></div>
    <span style="color:#64748b;font-weight:700;">→</span>
    <div style="background:#ffd6a5;color:#78350f;padding:8px 12px;border-radius:6px;border:1px solid #f59e0b;font-weight:600;">dequant<div style="font-weight:400;font-size:10px;">× sx · sw</div></div>
    <span style="color:#64748b;font-weight:700;">→</span>
    <div style="background:#a0c4ff;color:#1e3a8a;padding:8px 12px;border-radius:6px;border:1px solid #60a5fa;font-weight:600;">Y &nbsp;BF16</div>
  </div>
  <div style="margin-top:12px;font-size:11px;color:#64748b;border-top:1px dashed #e2e8f0;padding-top:10px;">
    Between linears: <span style="background:#ffadad;color:#7f1d1d;padding:1px 7px;border-radius:4px;font-weight:600;">softmax</span> <span style="background:#ffadad;color:#7f1d1d;padding:1px 7px;border-radius:4px;font-weight:600;">SwiGLU</span> <span style="background:#ffadad;color:#7f1d1d;padding:1px 7px;border-radius:4px;font-weight:600;">LayerNorm</span> — none run in INT8, so the BF16 hand-off is unavoidable.
  </div>
</div>


The matmul is the only op that runs in 8-bit. Softmax, SwiGLU, and normalization need floating-point dynamic range, so every linear hands its output back as BF16 — the next linear re-quantizes from scratch. The dequant is not wasted work; it is the boundary between an INT8 island and the FP ops that surround it.


> **About the perplexity numbers in this post.** The accuracy results come from a *fake-quant* W8A8 module — quantize → dequantize → `F.linear` in BF16. That is the same numerical state a real INT8/FP8 tensor-core kernel produces (modulo accumulator precision), so PPL is faithful, but the "actual throughput" payoff requires a real low-bit matmul kernel (`torch._int_mm`, marlin, FP8 cuBLAS) that this lab does not benchmark. We measure accuracy here; throughput is a separate exercise.
{: .prompt-info }


Weights are quantized **once, offline**, per group of 128 columns. Activations are quantized **dynamically at runtime** because their range depends on the input — there is no calibration value you can bake in.


<div style="background:#f1f5f9;border:1px solid #e2e8f0;border-left:3px solid #bdb2ff;border-radius:0 6px 6px 0;padding:14px 18px;margin:16px 0;font-family:system-ui,sans-serif;">
  <div style="font-size:10px;font-weight:700;text-transform:uppercase;letter-spacing:.07em;color:#4c1d95;margin-bottom:8px;">Formula · Symmetric INT8</div>


$$
s = \frac{\max|t|}{127}, \qquad t_q = \mathrm{clamp}\!\left(\mathrm{round}\!\left(\frac{t}{s}\right),\, -127,\, 127\right)
$$


  <div style="font-size:12px;color:#64748b;margin-top:10px;line-height:1.7;">
    $t$ — a weight group (128 values) or an activation slice · $s$ — fp32 scale · grid is <strong>uniform</strong> on $[-127, 127]$ · symmetric ⇒ zero-point $= 0$, so dequant is a single multiply.
  </div>
</div>


```python
def quantize_w8_groupwise_sym(W, group_size=128):
    W_grouped = W.float().view(out_features, n_groups, group_size)
    scales = (W_grouped.abs().amax(dim=2) / 127.0).clamp(min=1e-8)
    q = torch.clamp(torch.round(W_grouped / scales.unsqueeze(2)), -127, 127)
    return q.to(torch.int8), scales        # weights: once, offline


def quantize_activation_sym(x, mode="per_token"):
    scale = (x.abs().amax(dim=-1, keepdim=True) / 127.0).clamp(min=1e-8)
    return torch.clamp(torch.round(x / scale), -127, 127).to(torch.int8), scale
```


The runtime module wires those two helpers together. We replace every `nn.Linear` in the model with a `W8A8Linear` that stores INT8 weights as buffers, quantizes the activation on every forward, dequantizes both, and runs the matmul:


```python
class W8A8Linear(nn.Module):
    """Per-group INT8 weight, dynamic INT8 activation. Dequant-then-matmul."""


    def __init__(self, linear, group_size=128, act_quant="per_token"):
        super().__init__()
        self.group_size, self.act_quant = group_size, act_quant
        q, scales = quantize_w8_groupwise_sym(linear.weight.data, group_size)
        self.register_buffer("w_int8", q)        # stored once, offline
        self.register_buffer("w_scale", scales)
        self.bias = linear.bias


    def forward(self, x):
        x_q, x_scale = quantize_activation_sym(x, mode=self.act_quant)  # dynamic
        x_fp = x_q.float() * x_scale                                    # dequant act
        w_fp = self._dequant_weight()                                   # dequant W
        out  = F.linear(x_fp, w_fp, self.bias)                          # BF16/FP32 matmul
        return out.to(x.dtype)


# patch every nn.Linear in the decoder layers
for name, mod in model.named_modules():
    for child_name, child in list(mod.named_children()):
        if isinstance(child, nn.Linear) and "lm_head" not in child_name:
            setattr(mod, child_name, W8A8Linear(child, group_size=128, act_quant="per_token"))
```


> **Weights are static, activations are not.** You can quantize a weight matrix offline because it never changes. You cannot pre-quantize activations — their magnitude is set by whatever token just arrived, so the scale must be recomputed every forward pass.
{: .prompt-info }

## 2. Activation granularity: per-tensor vs per-token


*Why does one scale per tensor turn an 8.93-PPL model into a 21.44-PPL one?*

Same weights, same 8 bits, same INT8 grid — the only difference is **how many activation scales** you compute. Per-tensor uses one scale for the whole `[T × H]` tensor; per-token uses one scale per row. That single change is the gap between a broken model (+12.50 PPL) and a deployable one (+0.25 PPL).


The cause is the outlier problem. About 1% of activation channels carry magnitudes ~100× the median. A single per-tensor scale is set by that outlier, so the other 99% of values collapse into a handful of INT8 levels near zero.


<div style="background:#f1f5f9;border:1px solid #e2e8f0;border-left:3px solid #6366f1;border-radius:0 6px 6px 0;padding:14px 18px;margin:16px 0;font-family:system-ui,sans-serif;">
  <div style="font-size:10px;font-weight:700;text-transform:uppercase;letter-spacing:.07em;color:#6366f1;margin-bottom:8px;">Formula · Scale count</div>


$$
s_{\text{per-tensor}} = \frac{\max_{i,j}|X_{ij}|}{127} \quad\text{(1 scalar)} \qquad
s_{\text{per-token}}[t] = \frac{\max_{j}|X_{tj}|}{127} \quad\text{(T scalars)}
$$


  <div style="font-size:12px;color:#64748b;margin-top:10px;line-height:1.7;">
    Per-token isolates each row's outlier into that row's own scale — a quiet token is no longer punished for a loud neighbor's outlier. Cost: $T$ scalars instead of 1, negligible vs the matmul.
  </div>
</div>


> **Per-token is the W8A8 default.** It costs one extra reduction per row and recovers almost all the accuracy. Reach for per-tensor only when the hardware kernel demands a single scalar scale — and then read sections 3 and 4 first.
{: .prompt-tip }

INT8 and FP8 store the same number of distinct values (256), but they place them differently. INT8 spreads its levels evenly. FP8-E4M3 spends 4 exponent bits to pack levels densely near zero and stretch them toward its ±448 max — the same long-tailed shape that weights and activations actually have. **The grid shape, not the bit count, is what changes.**


### How the log spacing is built


A byte of FP8-E4M3 has the same fields as any IEEE float — sign, exponent, mantissa — just sized down:


<div style="display:flex;align-items:center;gap:6px;justify-content:center;margin:14px 0;font-family:ui-monospace,monospace;font-size:13px;flex-wrap:wrap;">
  <div style="background:#a0c4ff;color:#1e3a8a;padding:8px 14px;border-radius:6px;border:1px solid #60a5fa;font-weight:700;">S<div style="font-size:10px;font-weight:400;">1 bit</div></div>
  <div style="background:#fdffb6;color:#713f12;padding:8px 14px;border-radius:6px;border:1px solid #facc15;font-weight:700;">E E E E<div style="font-size:10px;font-weight:400;">4 bits — picks binade</div></div>
  <div style="background:#caffbf;color:#14532d;padding:8px 14px;border-radius:6px;border:1px solid #86efac;font-weight:700;">M M M<div style="font-size:10px;font-weight:400;">3 bits — splits binade into 8</div></div>
</div>


<div style="background:#f1f5f9;border:1px solid #e2e8f0;border-left:3px solid #6366f1;border-radius:0 6px 6px 0;padding:14px 18px;margin:16px 0;font-family:system-ui,sans-serif;">
  <div style="font-size:10px;font-weight:700;text-transform:uppercase;letter-spacing:.07em;color:#6366f1;margin-bottom:8px;">Formula · normal FP8-E4M3 value</div>


$$
v = (-1)^s \cdot 2^{\,e-7} \cdot \left(1 + \frac{m}{8}\right), \qquad e \in [1,15],\ m \in [0,7]
$$


  <div style="font-size:12px;color:#64748b;margin-top:10px;line-height:1.7;">
    Bias 7 means $e=7$ encodes the binade $[1,2)$. The mantissa places 8 evenly-spaced ticks <em>inside</em> that binade — but each step up in $e$ doubles both the binade's width and its tick spacing. Spacing is uniform <em>within</em> a binade, exponential <em>across</em> binades. (E4M3FN reserves the all-ones encoding for NaN, which is why the max is $1.75 \cdot 2^{8} = 448$, not 480.)
  </div>
</div>

The step size grows by 512× from the smallest normal binade to the largest, but the 8 mantissa slots split each binade the same way — so the **maximum relative error stays ~1/16 ≈ 6%** at every magnitude. INT8's absolute step is fixed at $\max\lvert t\rvert /127$; near zero its relative error explodes toward 100%, while at the max it's ~0.4%. FP8 trades that for a flat ~6% across the whole range. For weights and activations whose magnitudes span several orders of magnitude inside a single tensor, flat relative error is what you actually want.


(Below $2^{-6} = 0.0156$, normal binades run out and FP8 switches to *subnormals* — a single fixed step of $2^{-9} \approx 0.00195$ all the way down to zero. That's why the 0.001 row in the table below rounds to 0.00195: it landed in the subnormal region, which trades the constant-relative-error guarantee for the ability to represent values smaller than the smallest normal binade.)

Rounding is implicit in the cast — you scale into the ±448 window, then `.to(torch.float8_e4m3fn)` snaps to the nearest representable level. No explicit `round`/`clamp` grid math like INT8.


```python
FP8_MAX = torch.finfo(torch.float8_e4m3fn).max   # 448.0


def quantize_w8_groupwise_fp8(W, group_size=128):
    scales = (W_grouped.abs().amax(dim=2) / FP8_MAX).clamp(min=1e-8)
    W_scaled = (W_grouped / scales.unsqueeze(2)).clamp(-FP8_MAX, FP8_MAX)
    return W_scaled.to(torch.float8_e4m3fn), scales   # cast = round
```

> **The 0.001 row is a warning, not a win.** FP8's relative error explodes to 95% for values far below the scale's max — anything much smaller than the group's absmax rounds badly. The log grid helps the bulk, not the deep tail. Group scaling is what keeps that tail in range.

The matmul $Y = X W^\top$ is unchanged if you scale activation channel $c$ down by $s_c$ and the matching weight column up by $s_c$: $X \,\mathrm{diag}(s)^{-1} \cdot \mathrm{diag}(s)\, W^\top = X W^\top$. SmoothQuant picks $s$ to drain the outlier out of the activation (which you quantize at runtime, per-tensor) and into the weight (which you quantize once, offline, where you can afford per-group precision).


Calibration measures just how lopsided the activations are. On Qwen3-4B's first attention layer:

    <div style="font-size:11px;color:#7f1d1d;text-transform:uppercase;">outlier ratio</div>
    <div style="font-size:24px;font-weight:800;color:#7f1d1d;">9.2×</div>

<div style="background:#f1f5f9;border:1px solid #e2e8f0;border-left:3px solid #6366f1;border-radius:0 6px 6px 0;padding:14px 18px;margin:16px 0;font-family:system-ui,sans-serif;">
  <div style="font-size:10px;font-weight:700;text-transform:uppercase;letter-spacing:.07em;color:#6366f1;margin-bottom:8px;">Formula · Smooth scale (α = 0.5)</div>


$$
s_c = \frac{\big(\max_i |X_{ic}|\big)^{\alpha}}{\big(\max_j |W_{cj}|\big)^{1-\alpha}}, \qquad
Y = \underbrace{(X\,\mathrm{diag}(s)^{-1})}_{\text{easy to quantize}}\;\underbrace{(\mathrm{diag}(s)\,W^\top)}_{\text{fused offline}}
$$


  <div style="font-size:12px;color:#64748b;margin-top:10px;line-height:1.7;">
    $\alpha = 0.5$ — split the difficulty evenly between activation and weight. The per-channel $s$ is folded into the weights at build time, so inference only divides the activation by $s$ (one cached vector) before its per-tensor INT8 quant.
  </div>
</div>


```python
def _compute_smooth_scale(act_scale, weight, alpha=0.5):
    w_scale = weight.float().abs().amax(dim=0).clamp(min=1e-8)
    act_scale = act_scale.float().clamp(min=1e-8)
    return (act_scale.pow(alpha) / w_scale.pow(1 - alpha)).clamp(min=1e-8)


# offline: fuse s into weights, then quantize the smoothed weight to INT8
s = _compute_smooth_scale(act_scales[name], linear.weight.data, alpha=0.5)
W_smooth = linear.weight.data.float() * s.unsqueeze(0)
q, scales = quantize_w8_groupwise_sym(W_smooth, group_size)   # store q, scales, s
```


The result: smoothing alone takes per-tensor INT8 from a broken +12.50 PPL down to **+0.15** — within noise of the per-token result, while keeping the single cheap per-tensor activation scale the hardware wants.


> **SmoothQuant trades a runtime cost for an offline one.** The outlier doesn't vanish — it moves into the weights, where you already pay for per-group quantization offline. Activation quant stays per-tensor and cheap. That asymmetry is the whole point.
